# Obsidian-Wiki-LLM

**🌐 [English](README.en.md) | 中文**

> 这不是笔记软件,是让 AI 当馆员的知识操作系统——人只负责往里丢东西和提问题,AI 负责摘要、归档、建立关联、保持新鲜度。理论基础是 Karpathy 的 LLM Wiki 思想:普通 RAG 每次提问都重新推导一遍然后忘掉,这套系统把每次推导的结果编译一次、持续复用,知识像滚雪球一样越用越厚。

灵感来自两篇 gist:

- [Andrej Karpathy — LLM Wiki v1](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f):别把 LLM 当查询器,把它当馆员;原始资料是「源代码」,LLM 持续编译出结构化 Wiki。
- [rohitg00 — LLM Wiki v2](https://gist.github.com/rohitg00/2067ab416f7bbe447c1977edaaa681e2):补上生命周期治理 —— 认知状态、显式替代、冲突复核。

本项目是这两套思想在 **Obsidian + Claude Code** 上的实际落地方案,经过真实使用两个月以上迭代打磨,核心升级是把流程从"写在文档里靠 AI 记得做"变成"六个可执行斜杠命令"。

## 架构:三层 + 四个动作循环

**存什么(三层)**

| 层 | 存的是 | 规则 |
|----|--------|------|
| 原始层 | PDF、网页剪藏、扫描件 | 不可改写,是事实的最终裁判 |
| 编译层 | AI 写的摘要页、项目笔记、每日记录 | 会持续生长,是真正被检索的部分 |
| 系统层 | 一份规则文件 + 六个可执行命令 + git 版本历史 | 管这套系统怎么运转 |

**怎么转(四环)**:捕捉 → 入库 → 调用 → 进化 → 回到捕捉,循环往复。

```
捕捉(人) → 入库 /ingest → 调用 /ask /daily → 进化 /crystallize /lint /review → 回到捕捉
```

## 一、知识怎么入库

人的动作极简:看到有价值的东西,直接丢进一个「收件箱」文件夹,不用整理格式。

AI 的动作(触发一个入库命令后自动做):

1. 读原文,判断属于哪个知识领域
2. 提炼成结构化页面 —— 核心要点一两句话讲清本质,关键数据/公式单独列出,重要结论标注来源出处(不脑补原文没有的东西)
3. 自动关联到 2-5 篇已有相关笔记,建立双向链接
4. 更新总索引,在演进日志里留一条记录
5. 原始文件如果是不可再生的(比如扫描版 PDF),移到专门的原始资料区并记录哈希值存档;普通剪藏处理完可以清空,因为 git 已经留了历史

核心原则:**入库不是复制粘贴,是提炼、重组、建立联系** —— 把「知道」变成「有结构的理解」。

## 二、知识怎么整理与保鲜

- **每日**:第一次打开时,AI 自动建当天记录、把昨天没做完的待办带过来、扫一遍收件箱有没有堆积
- **每周复盘**:AI 翻这一周的日常记录,找反复出现的想法,提议把它们提炼成正式的知识页;检查进行中的项目有没有停滞;清理「知识空白清单」——遇到过但库里还没有的话题
- **每月体检**:全库扫一遍 —— 有没有链接指向不存在的笔记、有没有孤立的页面没人链接它、有没有过时太久没更新的重要内容、有没有两篇笔记说法互相矛盾。矛盾之处**不会自动裁决谁对谁错**,标记出来等人复核
- **会话结晶**:每次深度讨论/调试/做决策后,把问题、证据、最终结论、成立的边界条件、被否掉的方案存成一页——避免「这个上次想清楚了但现在忘了」

## 三、需要的时候怎么检索总结

这是整个系统最关键的一条规矩,回答任何问题前必须走完:

```
① 先查索引 —— 不是凭空回答,是先去总索引和分类索引里定位可能相关的笔记
② 打开原文读 —— 真正打开命中的笔记读内容,不靠记忆瞎猜
③ 综合输出 —— 基于读到的内容整理回答,每个关键结论标注是从哪篇笔记来的
④ 有价值就存回去 —— 如果这次问答产生了库里没有的新洞见,存成新页面,库越用越大
⑤ 诚实兜底 —— 库里确实没有相关内容,就明确说「知识空白」,而不是拿大模型的通用知识冒充是你自己记录的内容
```

好处是**答案可溯源**——不是模型编的,是能指回具体笔记的。库里确实没有的东西,AI 会说清楚这是"额外补充"而非你的知识库内容。

## 四、怎么构建一套属于自己的

不需要从零设计,三步就能搭起来:

1. **搭骨架**:几个功能分区(收件箱 / 进行中项目 / 长期领域 / 知识库主体 / 归档 / 每日记录),外加一份说明系统怎么运转的规则文件
2. **把流程做成命令,不要只写在文档里**——这是最容易被忽略但最重要的一步。规则写在文档里,AI 隔几次会话就会"忘了走流程";做成可以直接调用的固定命令,流程才能稳定执行、不受换会话换模型影响
3. **用 git 接管版本历史**——每次大改动前存一个检查点,AI 才敢大胆维护,你才敢大胆放手让它做,出问题随时能回滚

几条容易踩的坑:

- **别一上来就把系统做复杂**——目录别套太深,规则别写两份互相不同步的文档,先跑起来,复杂功能等真实需求出现再加
- **别太早上向量数据库/语义搜索**——几十上百篇笔记规模,简单的索引+双向链接又快又好用,规模真正大了(几百篇以上)才需要更复杂的检索
- **重要结论必须能查到出处**——这是防止 AI 一本正经编造内容的生命线
- **人的负担要降到最低**——人只做「丢东西」和「提问题」两件事,格式、分类、建链接这些苦活全交给 AI,人的动作一旦变复杂,系统很快就会被弃用

## 快速开始

1. 新建一个空文件夹,用 Obsidian 打开作为 vault,在其中启动 Claude Code(或任何能读写文件的 AI agent)。
2. 打开 [`BOOTSTRAP.md`](BOOTSTRAP.md),把里面的提示词整段复制发给 AI。
3. 回答 AI 关于你自己的几个问题(称呼、身份、当前重心、建库初衷)。
4. 系统建成后,日常只需要:每天 `/daily` 开馆、随手把东西丢进 `00.Inbox/`、攒几篇跑 `/ingest`、有问题 `/ask`、想清楚一件事 `/crystallize`、每周 `/review`、每月 `/lint`。

更完整的原理讲解见 [`GUIDE.md`](GUIDE.md)。

## 与传统 RAG 的区别

| 维度 | 普通 RAG | 本方法(LLM Wiki) |
|------|---------|------------------|
| 知识存储 | 向量数据库 | Obsidian Markdown 文件 |
| 检索方式 | 语义相似度召回 | 索引 → 双链导航 |
| 知识复用 | 每次重新拼接上下文 | 编译一次,持续复用 |
| 可读性 | 对人不友好(向量) | 人和 AI 都可直接读 |
| 规模门槛 | 需要嵌入 API + 向量库 | 只需 Claude Code,小库无需向量化 |

本方法不排斥 RAG —— wiki 页超过 ~200 篇、索引定位明显失手时,再引入本地混合检索(BM25 + 向量)作为补充。
