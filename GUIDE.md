## 一、第一性原理:知识库到底要解决什么

大多数人的笔记库最终变成收藏夹:文章剪进来再也不看,答案问完就消失在聊天记录里。知识**被存了,但从未被用**。

从头推,个人知识库只需要回答四个问题:

| 问题 | 传统做法的失败点 | 本方法的答案 |
|------|----------------|-------------|
| **存什么?** | 囤原文,越囤越不敢看 | 原文只是「源代码」,存的是 AI **编译后的理解**(摘要+关联+结论) |
| **怎么找到?** | 靠记忆和全文搜索 | 索引优先:AI 回答前先查 `_index` → 读原文 → 再综合,不脑补 |
| **怎么被用?** | 没有机制,全凭自觉 | 每天 `/daily`、每问 `/ask`、每周 `/review` —— 调用是流程,不是愿望 |
| **怎么不腐化?** | 笔记只增不改,悄悄过时 | `/lint` 体检 + `/crystallize` 结晶 + supersession 显式替代 + git 历史 |

核心转变(Karpathy):**别把 LLM 当查询器,把它当馆员。** 人负责选来源、提问题;AI 负责摘要、归档、建链、保鲜 —— 记账的苦活全给机器。

## 二、架构:三层四环

**三层** 回答「存什么」:

```
L0 原始层(不可变):PDF、剪藏、OCR 文本,带 SHA-256 —— 事实的最终裁判
L1 编译层(会生长):wiki 页、项目笔记、daily —— AI 持续编译维护
L2 系统层(管运行):CLAUDE.md 宪法 + memory.md 补丁 + 命令 + git
```

**四环** 回答「怎么转」,每一环都是可执行命令而非口号:

```
捕捉(人,零格式) → 入库 /ingest → 调用 /ask /daily → 进化 /crystallize /lint /review → 回到捕捉
```

与普通 RAG 的区别一句话:**RAG 每次重新推导然后忘掉;LLM Wiki 编译一次持续复用。** 综合结果、交叉链接、矛盾标记都持久保存,知识随使用复利增长。(小库连向量数据库都不用 —— 索引+双链更便宜也更可读;wiki 页 >200 再上混合检索。)

## 三、系统的三份关键文件

1. **CLAUDE.md(宪法)**:唯一系统规则。八节结构见 [BOOTSTRAP.md](BOOTSTRAP.md) 第 2 步。写作要领:**够用就好** —— 你是谁、检索协议、写入规范、红线,其余交给 memory 慢慢长。
2. **memory.md(补丁)**:AI 在使用中持续记录你的偏好,冲突时覆盖宪法。这让系统**越用越懂你**,而宪法保持稳定。
3. **.claude/commands/(灵魂)**:常见的失败模式 —— 写在手册里的流程,AI「有时记得做」;做成斜杠命令后,流程变成**按键可执行**,换会话、换模型都不退化。规则不靠记忆,靠命令。

## 四、六级成长路线(从零到成熟)

按需逐级加,**不要一次建满**:

| 级 | 加什么 | 何时加 |
|----|--------|--------|
| 1 最小可用 | 目录骨架 + CLAUDE.md + _index/_log | 第一天([BOOTSTRAP.md](BOOTSTRAP.md) 全包) |
| 2 生命周期 | epistemic_status、source 溯源、superseded_by | 库里出现第一次「新旧说法打架」时 |
| 3 命令化 | 六命令 + git | 发现 AI「忘了走流程」时(建议第一天就上) |
| 4 复用闭环 | /review 周复盘 + 知识空白清单 | 感觉「存了很多但没用起来」时 |
| 5 规模化 | BM25+向量混合检索 | wiki 页 >200 或索引定位明显失手时 |
| 6 多 agent | AGENTS.md 统一约束多个 AI 工具 | 用第二个 AI 工具进库时 |

## 五、避坑指南(实战踩过的坑)

1. **过度工程是头号杀手**。曾经一天建出多层 schema 文档 + 深层嵌套目录,第二天全部推倒重来 —— 结构必须直观简洁:目录 ≤3 层,系统文档只有一份,规则写一处。
2. **别造重复文档**。如果同时给 Claude 和其它 AI 工具写规则,很容易变成两份全文副本、改一处漏一处;正确做法是「一份完整 + 一页指针」。任何信息只允许有一个权威出处。
3. **警惕虚假精度**。给每页标 0.0-1.0 置信度并自动衰减听起来很酷,实际没人校准 —— 离散认知状态(fact/hypothesis/decision…)比小数诚实得多。
4. **AI 写入必须留证据链**。重要结论必须有 `source`;精确数字/公式回原文核对;冲突不自动裁决(新的不一定对)。这是防「一本正经胡说八道」的生命线。
5. **别提前上向量库**。几十页的库,索引+双链检索又快又可读;部署向量库是把简单问题复杂化。
6. **git 越早越好**。有了回滚安全网,AI 才敢大胆维护,你才敢大胆授权。
7. **降低捕捉摩擦到零**。人只负责往 Inbox 丢东西,格式、归类、建链全是 AI 的事 —— 捕捉一旦有门槛,系统就死了。

## 六、动手构建你自己的库

1. 打开 [BOOTSTRAP.md](BOOTSTRAP.md),复制提示词到空 vault 的 Claude Code 里,回答几个采访问题 → 系统建成。
2. 第一周只做三件事:每天 `/daily`、往 Inbox 丢东西、攒几篇跑 `/ingest`。
3. 第二周开始 `/ask` 提问(体会「先查库再生成」的差别)和 `/review` 周复盘。
4. 系统哪里不顺手,直接告诉 AI —— 它会写进 memory.md,越用越合身。
